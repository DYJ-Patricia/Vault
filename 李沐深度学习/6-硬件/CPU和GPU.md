# CPU
## 提升 CPU 利用率 Ⅰ

- 在计算 a+b 之前，需要准备数据
    
    - 主内存 -> L3 -> L2 -> L1 -> 寄存器（数据只有进入寄存器时才能参与运算）
    - L1 访问延时：0.5 ns
    - L2 访问延时：7 ns (14 × L1)
    - 主内存访问延时：100 ns (200 × L1)
    
- 提升空间和时间的内存本地性
    - 时间：重用数据使得保持它们在缓存里
    - 空间：按序读写数据使得可以预读取

### 样例分析

- 如果一个矩阵是按行存储，访问一行会比访问一列要快
- CPU 一次读取 64 字节（缓存线）
- CPU 会 “聪明的” 提前读取下一个（缓存线）

## 提升 CPU 利用率 II

- 高端 CPU 有几十个核
    
    - EC2 P3.16xlarge: 2 Intel Xeon CPUs, 32 物理核
    
- 并行来利用所有核
- 超线程不一定提升性能，因为它们共享寄存器


### 样例分析

- 左边比右边慢（python）

``` python
for i in range(len(a)):
    c[i] = a[i] + b[i]
```

``` python
c = a + b
```

- 左边调用 n 次函数，每次调用有开销
- 右边很容易被并行（例如下面 C++ 实现）

``` cpp
#pragma omp for
for (i=0; i<a.size(); i++) {
    c[i] = a[i] + b[i]
}
```

- 所以尽量用多核
# GPU

- 一个绿点即核能算一个指令，但有上千个
# CPU vs GPU

|指标|CPU（一般 / 高端）|GPU（一般 / 高端）|
|---|---|---|
|# 核|6 / 64|2K / 4K|
|TFLOPS|0.2 / 1|10 / 100|
|内存大小|32 GB / 1 TB|16 GB / 32 GB|
|内存带宽|30 GB/s / 100 GB/s|400 GB/s / 1 TB/s|
|控制流|强|弱|
- 内存不贵，显存贵

## 提升 GPU 利用率

- 并行
    
    - 使用数千个线程
    
- 内存本地性
    
    - 缓存更小，架构更加简单
    
- 少用控制语句
    
    - 支持有限
    - 同步开销很大

# CPU/GPU 带宽

- CPU 与内存带宽：**30 GB/s**
- CPU 与 GPU 带宽（PCIe 3.0 16x）：**16 GB/s**
- GPU 内部显存带宽：**480 GB/s**
- 核心建议：**不要频繁在 CPU 和 GPU 之间传数据**，因为带宽有限，且同步开销大。
